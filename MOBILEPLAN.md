# MOBILEPLAN.md — CanonFlow → Android & iOS

> Thesis: the mobile app is an **emitted edge**, not a new codebase philosophy.
> F# stays the engine room; Kotlin and Swift are theorems written in the
> language of their readers. No MAUI (2026 regression storm, Fabulous.Maui
> ON-HOLD), no Xamarin (EOL May 2024). Native clients consuming radiated
> contracts — which makes this plan simultaneously the Phase 3 Kotlin/Swift
> story dogfooded, using Layam Academy as the sample app.

---

## 0. The pipeline (one diagram, both platforms)

```
Postgres (Layam)                                   CanonFlow owns this half
  └─ canonflow introspect ──► Canon (lattice + Refined + provenance)
       ├─ canonflow contracts --openapi  ──► openapi.yaml   (source of truth)
       ├─ canonflow contracts --ts       ──► validate.ts    (exists today)
       └─ [NEW] canonflow contracts --validators kotlin|swift
                                          ──► Validators.kt / Validators.swift
------------------------------------------------------------------
openapi.yaml ─ openapi-generator ─► Kotlin models (kotlinx.serialization)
             └ openapi-generator ─► Swift models (Codable)      stock tools
Validators.kt/.swift ─► pure functions, zero deps, mirror of the lattice
```

Two artifacts per platform, deliberately split:
- **Models** come from stock OpenAPI generators (ADR-005 — never hand-roll).
- **Validators** come from a new thin CanonFlow emitter, because constraint
  semantics (two-bound ranges, exclusive bounds, decimal precision) is the
  part stock generators fumble — and the part that is our whole thesis.

## 1. What CanonFlow must add (the only framework work in this plan)

**T-M1 · Validator emitters for Kotlin + Swift** — extend the Fable
Transpiler's walk (same lattice fold, different render):
`Range` → `value > 18.toBigDecimal()` (Kotlin `BigDecimal`, Swift `Decimal` —
never Double: same reasoning as the FParsec pfloat fix), `And/Or/Not` →
boolean expressions, `Opaque` → emitted as a listed *unchecked* constraint in
a header comment + a `warnings` list at runtime. Output: one pure file per
platform, zero dependencies, provenance comments per function
(`// derived-from: batches.fee_monthly via CHECK (fee_monthly >= 500 AND ...)`).
Fidelity per function, per T4.

**T-M2 · Tri-runtime agreement test** — extend AgreementTests: random
lattices + values, evaluate in F#, Node, **and Kotlin (via kotlinc -script
in CI)**; Swift arm runs on the macOS CI lane. The Phase 3 gate generalized:
one predicate, N runtimes, identical verdicts. This test is the plan's
falsifier — if it can't go green, native emitters retreat to labs/ and
mobile uses JSON-Schema-driven validation instead (fallback pre-agreed).

**T-M3 · `canonflow scaffold --compose`** — port ScaffoldForms' React walk
to Jetpack Compose: one `@Composable` per table (TextField per column,
keyboard type from data type, error text from validator + provenance).
Scaffold, not product — humans style it after.

## 2. Android (first, and alone until it ships)

Why first: India's install base is overwhelmingly Android; Compose is boring
in the way MAUI currently is not; and Kotlin is CanonFlow's declared first
contract target.

**App: Layam Academy Admin** (the dogfood domain — enrollment, batches,
payments, exams).

Module layout:
```
app/            Compose UI, navigation, DI (Hilt or manual — keep thin)
contracts/      GENERATED ONLY: openapi-generator models + Validators.kt
                — regenerated by Gradle task `canonflowSync`, never edited,
                CI fails if `git diff` after regeneration is non-empty
                (drift detection for clients — Drift's philosophy at the edge)
data/           Ktor client + SQLDelight offline cache
domain/         thin use-cases; enrollment rules = call generated validators
```

Low-level decisions, pre-made:
- **kotlinx.serialization** over Moshi/Gson (generator flag `library=jvm-ktor`,
  matches Ktor).
- **SQLDelight** for offline cache — schema-first like everything else here;
  its .sq files are *derived from openapi.yaml* by a small generator step, so
  even the cache schema traces to Postgres constraints.
- **Form state:** each field holds `value: String` + `verdict: Verdict` where
  `Verdict = Valid | Invalid(reason, provenance)` — the generated validator
  runs on every change; the error text can *show the CHECK constraint* (the
  mobile "oh." moment: a teacher sees "fee must be ≥ ₹500 — school policy,
  enforced by the database" on their phone).
- **Version skew:** every contracts bundle carries the schema fingerprint
  (hash of canonical model). App sends it as a header; server replies 409 on
  mismatch → app prompts update. Cheap, prevents the classic stale-client
  constraint fight.
- Min SDK 26, single-activity, no fragments, no XML.

Definition of done (falsifiable): a stranger with Android Studio runs
`docker compose up` + `./gradlew installDebug` and enrolls a student in
under 30 minutes; entering ₹499 is rejected *offline* with provenance shown;
T-M2 agreement green in CI.

## 3. iOS (second, contracts-first, smallest possible)

Trigger to start: Android DoD met AND one real request for iOS exists.
Until both, iOS is a directory with a README.

- Models: openapi-generator swift6 target (Codable, async/await URLSession).
- Validators.swift from T-M1 — `Decimal`, exclusive-bound logic mirrored,
  same provenance comments.
- UI: SwiftUI, thin — mirror the Compose screens, don't innovate twice.
- Agreement: macOS CI lane runs the Swift arm of T-M2.
- **Explicit non-goal:** no shared UI framework, no KMP *yet* — evaluate
  Kotlin Multiplatform only when both apps exist and the duplication is
  *measured*, not feared. (KMP shares the wrong layer for us: our shared
  layer is the contract, already shared at zero cost.)

## 5. Risks (eyes open)
- **Decimal semantics across four runtimes** (F#/JS/Kotlin/Swift) is the
  hardest agreement problem — JS has no decimal; the TS validator must use
  string-safe comparison for money fields or the tri-runtime test *will*
  catch it. This is a feature: the bench exists to find exactly this.
- **Emitter sprawl**: T-M1 is two render functions, not two new modules. If
  it exceeds ~300 lines per language, stop — that's the native-emitter
  labs/ boundary being crossed.
- **The app becoming a product**: Layam Admin is a dogfood artifact. Feature
  requests get `post-v1` labels like everything else.

*The phone in a parent's hand rejecting ₹499 offline, citing a Postgres CHECK
constraint it has never met — that is the whole thesis, four inches wide.*
